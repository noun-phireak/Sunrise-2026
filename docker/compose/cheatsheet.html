Foundation — what runs
01services

The unit of Compose — one container spec per service.

services: web: image: nginx

Service name = DNS hostname on the network. Bring up with up.

02image

Run a prebuilt image pulled from a registry.

image: redis:7-alpine

Always pin a tag — never rely on the moving :latest.

03build

Build an image from a Dockerfile instead of pulling.

build: context: . args: { V: "2.0" }

Only builds on first up. Force with --build.

Networking & exposure
04ports

Publish a container port to the host.

ports: - "8080:80" # HOST:CONTAINER

Left = host, right = container. Swapping them is the classic bug.

08networks

Service-to-service DNS by name; segment traffic.

networks: [backend]

Reach a peer at http://api:8080 — no ports needed internally.

Configuration
05environment

Set env vars inside the container.

environment: APP_NAME: "Sunrise"

Double the dollar: $$VAR passes a literal $ into the container.

06env_file

Load env vars into the container from a file.

env_file: [app.env]

Different from the special .env, which fills ${VARS} in the compose file.

16secrets / configs

Mount sensitive/bulky data as files, not env vars.

secrets: [db_password] # -> /run/secrets/db_password

Safer than environment: — env leaks via inspect & /proc.

Data
07volumes

Persist & share data — named volume or bind mount.

volumes: - data:/var/lib # named - ./site:/www:ro # bind

Named volumes survive down; remove with down -v.

Lifecycle & health
09depends_on

Startup order between services.

depends_on: db: condition: service_healthy

Plain form waits only for start; use the condition for ready.

10healthcheck

Test a service is actually ready, not just running.

healthcheck: test: ["CMD","wget","-qO-","localhost"] interval: 5s

Exit 0 = healthy. Shows as (healthy) in ps.

11restart

What to do when a container exits or the host reboots.

restart: unless-stopped

no · on-failure · always · unless-stopped. Not a healthcheck.

Runtime overrides
12command / entrypoint

Override the image's default command.

entrypoint: ["sh","-c"] command: ["echo hi"]

Prefer exec form (JSON) so SIGTERM reaches your process.

13scale / replicas

Run many containers of one service.

deploy: replicas: 3 # or: up --scale worker=3

A scaled service can't use a fixed host port — they'd collide.

14profiles

Optional services activated on demand.

profiles: ["debug"] # up --profile debug

No profile = always on. Activate with --profile or COMPOSE_PROFILES.

DRY & capstone
15extends / anchors

Reuse config across services — don't repeat yourself.

x-common: &base restart: on-failure web: <<: *base

Anchors = one file; extends can pull from another file.

99all-in-onecapstone

A realistic 4-service stack: web → api → db + cache.

# web (proxy) · api (built) # db (postgres) · cache (redis)

Combines networks, volumes, healthchecks, depends_on, secrets & profiles.
